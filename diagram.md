1. Event Created
- created by Admin or Vendor
- Event date and time
- City/Location
- Required Roles (Bartender, Server, Kitchen)
- Quantity Per Role

2. Labor Pool (Pre-Existing Staff)
- workers already exist in system
- Each worker has:
 - role type 
 - city
 - availability status

3. Invite New Staff (If Needed)
 - Admin sends invite by email 
 - staff completes onboarding
 - Role + City Assigned
 - Staff Enters "Idle/Available" state

4. Dispatch Board (Dashboard-Level)
 - Visible to all user types
 - Role-based permissions:
  - Admin: Full control
  - Vendor/Client: Event Specific visibility and interaction
  - Staff: personal job feed only
 - list of open roles per event
 - Status per role: unfilled/pending acceptance/filled



5. Send Job Request

 - Admin Selects:
  - Role Type
  - City
  - Event
 -System sends request to matching staff


6. Staff Receives Job Ping

 - delivered via 
  - mobile app notification
  - in app notification
 - includes event name, role, date, and time
 - accept/decline buttons


So if a staff member accepts then the path is first acceptance locks role, and staff assigned to event, event permissions granted automatically

So if a staff member declines then the path is staff returned to available pool, dispatch remains open, system ntofies next available staff



7. Staff attached to event
 - role marked as filled
 - appears on dispatch board
 - appears on event details

8. Event group chat
 - participants: admin, vendor/client, assigned staff
 - features
   - pre-event coordination
   - live event messaging
   - post-event follow up



9: 1:1 admin - staff chat
  - private communication
  -issue handling
  -special instructions

10. Live event mode
 - staff uses mobile app as walkie talkie
 - admin and vendor can message group
 - status updates visible in dispatch board
 - real-time coordination

11. Event closed
 - chat archived
 - staff released back to labor pool
 - event marked complete 
 - ready for next assignment
